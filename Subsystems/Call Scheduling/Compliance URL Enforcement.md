# Compliance URL Enforcement — Gong vs Native Provider Links

> [[_dashboard|← Team Hub]] · [[Use Cases/C - Cancel/C3 - Cancel On Resolution Change|UC-C3]] · [[Subsystems/Consent/Jump Page & DCP]]

> [!note] TL;DR
> A rep doesn't *have* to use the Gong add-on — but what happens when they use the native Zoom/Teams/Webex link instead depends entirely on the company's DCP `isEnforced` flag. `isEnabled=false` → nothing. `isEnforced=false` → call schedules, rep gets a warning email. `isEnforced=true` → **call is blocked**, not scheduled for recording.

---

## What triggers the check

`CheckCompliance` (`CallScheduler/.../validation/CheckCompliance.java`) is one validator in the scheduling pipeline. It runs every time a call is being evaluated for scheduling — calendar-sync, email invite, or manual schedule.

It reads the **Data Capture Profile (DCP)** for the call owner, then inspects every meeting URL in the event.

---

## The decision tree

```
DcpJumpPageSettings for call owner
        │
        ├─ null / isEnabled = false
        │       → nothing, call schedules normally
        │
        └─ isEnabled = true
                │
                All URLs are Gong jump-page URLs?  ← jumpPageUrlService.isJumpPageUrl(url)
                │
                ├─ yes → nothing, call schedules normally
                │
                └─ no (at least one URL is a raw provider link)
                        │
                        ├─ isInterview = true → skip enforcement (interviews exempt)
                        │
                        ├─ shouldSendNonCompliantCallsEmail = true
                        │       → enqueue warning email to rep
                        │         (includes their Gong jump-page link)
                        │
                        └─ isEnforced?
                                ├─ false → call schedules (warning email only)
                                └─ true  → Resolution.COMPLIANCE_ENFORCING
                                           call NOT scheduled for recording
```

---

## The three outcomes in plain terms

| DCP state | Rep uses Zoom directly | Result |
|---|---|---|
| `isEnabled = false` | Consent not configured | Call schedules normally |
| `isEnabled = true`, `isEnforced = false` | Non-Gong URL detected | Call schedules; rep receives a non-compliant URL warning email containing their correct Gong jump-page link |
| `isEnabled = true`, `isEnforced = true` | Non-Gong URL detected | **`Resolution.COMPLIANCE_ENFORCING`** — call is not scheduled for recording. Downstream: `SchedulingCallService#cancelExistingCallDueToResolutionChange` fires, `call-scheduling-updated` (`CANCEL`) produced |

---

## Interview exemption

If `context.isInterview()` is `true`, the whole check is skipped regardless of `isEnabled` or `isEnforced`. Recruiting interviews bypass consent URL enforcement entirely.

---

## What "is a Gong jump-page URL" means

`jumpPageUrlService.isJumpPageUrl(url)` checks whether the meeting URL matches the Gong consent-page URL pattern. A raw Zoom/Teams/Webex link (e.g. `https://zoom.us/j/...`) does **not** match.

The Gong link — generated by `JumpPageUrlService` and embedded in calendar invites by Call Scheduling — routes the participant through the consent page before redirecting to the provider. See [[Subsystems/Consent/Jump Page & DCP#How the URL reaches the participant]].

---

## The warning email

When `shouldSendNonCompliantCallsEmail = true` and a non-compliant URL is detected, `CheckCompliance` enqueues a `JumpPageEnableEnforceEmail` to the call owner. It includes:

- Whether enforcement is on or off (`isEnforced`)
- The rep's correct Gong jump-page URL (PMI or dynamic, depending on `linkType`)
- The `linkType` as a string

This is the notification path that nudges reps to switch their meeting link before enforcement is ever turned on.

---

## Where `isEnforced` is set

`isEnforced` lives on `DcpJumpPageSettings` (field `isEnforced: boolean`). It is set via the DCP settings API (`RecordingConsentApiServer`, `DcpConsentSettingsController`) — an admin-only action. When a company flips `isEnforced` from `false` to `true`, `DcpChangeManager` propagates the change to all users via the [[Subsystems/Consent/Use Cases/E - Propagate/E1 - Orchestrate Change Request|change-request state machine]], which includes running `CancelNonCompliantCallsAction` to cancel any already-scheduled calls that are now non-compliant.
